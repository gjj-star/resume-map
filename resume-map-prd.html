简历分析能力地图

Resume Map — 上传简历即生成可量化的岗位能力地图与补强路线
产品Resume Map（简历能力地图） 作者候选人 版本V0.1（MVP 规划） 日期2026-07-15 状态需求评审稿

1产品概述

1.1 背景与问题

求职者（尤其应届生 / 转岗者）在准备简历与面试时普遍面临三个断层：

  • 自我认知模糊：清楚"做过什么"，但说不清"相对目标岗位，我哪强哪弱"。
  • 表达与真实脱节：简历常写"熟悉 XX / 主导 XX"，面试被追问即露馅，缺乏"看起来会 → 真会"的校准机制。
  • 补强无路径：知道有短板，但不知道"闲时先补哪块、按什么顺序"，时间被低效消耗。

现有简历工具多停留在"排版美化 / 错别字检查 / 关键词匹配 ATS"，没有把简历映射成一套可量化、可对比、可行动的能力视图。这正是本产品的切入点。

1.2 产品定位

一句话定位
一款上传简历即生成「岗位能力地图」的轻量 Web 工具：用雷达图量化能力分布，标记面试风险点，并给出按优先级排序的补强路线——让求职者把"模糊的简历"变成"可行动的成长地图"。

1.3 核心价值

对求职者
看清能力水位与雷区，把零散经历结构化，面试有底气、补强有顺序。
对产品本身
验证"AI + 简历 + 可视化"是否真有需求，沉淀可复用的能力维度库与评估引擎。
对作者（作品集）
一个完整闭环的 AI 产品案例：从规划、原型到 PRD，展示产品方法论。

2目标与成功指标

2.1 业务目标

  • V0（本版范围）：用规则引擎跑通"上传 → 解析 → 能力地图"闭环，验证用户是否愿意为"能力地图"买单（哪怕只是觉得"有意思 / 有用"）。
  • 北极星指标（V0）：生成完整能力地图的报告数 / 周（而非 UV，避免被"好奇点一下"误导）。

2.2 成功指标（按阶段）

阶段核心指标目标（建议）
V0 规则验证周报告生成数、单次生成完成率≥ 50 份/周；完成率 ≥ 80%
V0 体验报告导出率、次日回访率导出 ≥ 40%；回访 ≥ 15%
V1 智能LLM 评估采纳率（用户不手动大改即视为采纳）≥ 70%
V2 留存多份简历对比使用率、补强计划完成率对比 ≥ 25%；计划完成 ≥ 30%
指标设计原则：重"产出"轻"流量"。本产品低频，看 UV 无意义；看"是否真生成了报告、是否导出/回访"才反映价值。

3用户与场景

3.1 目标用户

画像特征核心诉求
应届生 / 实习转校招（首要）经历少、方向未定、易堆砌"熟悉"看清能力水位、排雷、规划补强
转岗 / 跨行求职者有旧行业经验，缺目标岗证据把旧经历"翻译"成目标岗语言
自我提升者（延伸）暂不跳槽，想对标岗位查漏持续的能力体检

3.2 典型场景

  • 场景 A（投递前体检）：小白写完简历，上传生成地图，发现"AI 产品经理"所需的"技术概念理解"得分低 → 按 P0 建议补一份 PRD 练手。
  • 场景 B（面试备战）：地图标出 3 个风险点（如"C端项目措辞夸大"）→ 提前准备诚实话术，避免被抓。
  • 场景 C（规划充电）：空窗期对照"补强路线"按 P0/P1/P2 顺序学习，产出可写进简历。

3.3 用户故事（示例）

作为【应届求职者】，
我希望【上传简历并选择目标岗位后，得到一张能力雷达图+风险点+补强顺序】，
以便【在投递前知道自己的真实水位、排掉面试雷区、知道闲时该补什么】。

4功能需求

优先级采用 P0 必须（MVP）/ P1 重要（V1）/ P2 增强（V2+）。

编号功能说明优先级
F1简历上传与解析支持 PDF / Word 上传 → 文本提取（PDF 用 pypdf 已验证可行）。解析失败给出友好重试提示。P0
F2目标岗位选择下拉选择岗位（AI 产品经理 / 数据分析 / 前端 / 后端…），不同岗位加载不同能力维度库。P0
F3能力评估引擎核心。输出：①各维度评分 ②雷达图 ③维度点评。V0 用规则，V1 切 LLM。P0
F4风险点诊断识别"看起来会 / 前后不一致 / 精确数字无依据"等面试雷区，标记危险度并给改法。P0
F5补强路线建议按 P0/P1/P2 排序，给出"先补什么、怎么补、产出是什么"。P0
F6报告查看与导出网页渲染能力地图；支持"导出 PDF"（浏览器打印，已验证可用）与生成分享链接。P0
F7多份简历对比同一人不同版本 / 不同岗位对照，看能力迁移与差距。P1
F8面试 Q&A 生成基于风险点自动生成"面试官可能怎么问 + 你怎么答"的备战卡。P1
F9职业路径推荐基于当前能力，推荐 1-2 条可达的岗位进阶路径。P2
F10行业基准对比匿名聚合同岗位人群能力分布，给出"你处于 P 分位"。P2
MVP（V0）范围 = F1~F6，且 F3 评估引擎先用规则实现，不依赖大模型，做到零 API 成本、可离线、秒级响应。

5智能评估引擎设计（核心）

评估引擎是产品灵魂，也是"要不要接大模型"的分水岭。两种模式对应两个版本。

5.1 模式对比

维度规则引擎（V0）LLM 引擎（V1）
原理预置维度库 + 正则/关键词提取 + 固定评分函数大模型读取简历+维度定义，输出结构化 JSON
能力量化粗（命中关键词即加分）细（理解语义、上下文、强弱程度）
风险点仅规则命中（如"熟悉"无证据）可推理前后矛盾、夸大、无依据数字
成本零 API 成本按 token 计费（可控）
依赖无外部服务需 LLM API + 轻后端代理
上线速度快（天级）中（需 prompt 调优 + schema 校验）

5.2 V0 规则引擎方案

实现要点
  • 维度库：每个岗位一份 YAML/JSON 配置，含维度名、权重、关键词/正则、证据等级映射。
  • 评分：遍历简历文本，命中维度关键词累计得分 → 归一化到 0-100。
  • 风险规则：如正则 (熟悉|精通|主导).*(但无前后文证据)、数值+% 且附近无常量解释 等。
  • 局限：只能做"有没有"，做不出"好不好 / 上下文是否自洽"，因此 V0 必须明确标注"规则初评，仅供参考"。

5.3 V1 LLM 引擎方案

关键设计：结构化输出 + 后端代理
  • 后端代理：前端只把"简历文本 + 岗位维度定义"发给自有轻后端，后端持密钥调 LLM，密钥绝不进前端（安全红线）。
  • Prompt 约束：要求模型严格按固定 schema 输出 JSON，便于前端直接渲染雷达图。
  • 输出 Schema（示例）：
{
  "dimensions": [
    {"key":"数据分析","score":78,"level":"强","comment":"复购预测 AUC 0.8959，有量化结果"},
    {"key":"AI工具链","score":82,"level":"强","comment":"深度使用 WorkBuddy/Claude，独立接千问做小程序"},
    {"key":"技术概念理解","score":55,"level":"中","comment":"会 Vue3/小程序，但缺后端/部署/成本视角"}
  ],
  "risks": [
    {"level":"高","point":"C端项目措辞","fix":"补进项目列表并诚实描述规模"}
  ],
  "plan": [{"priority":"P0","action":"练一份完整 PRD","output":"可写进简历的 PRD 文档"}]
}
  • 容错：后端校验 JSON schema，缺失字段用规则兜底；模型失败降级到 V0 规则引擎，保证"不空场"。
  • 自带 Key（可选）：高级用户填自己的 API Key，走其账户计费，降低产品成本。

5.4 能力维度库（示例：AI 产品经理岗）

维度权重考察什么
数据分析能力20%是否用数据驱动决策、有无量化结果
AI 工具与落地20%是否真用过 AI 工具并产出
产品基本功20%PRD / 原型 / 需求拆解
技术概念理解15%是否懂 API/部署/成本，能和技术对话
业务理解15%行业/用户/商业逻辑
表达与诚实度10%简历是否自洽、有无夸大

7关键技术决策

7.1 落地方向 → Web 优先，不做 App
简历分析是低频、单次、重结果场景，用户不会为它装 App。Web 零安装、好分享（链接/二维码）、PC 看雷达图体验最佳。路线：Web MVP 验证 → 视情况做小程序（分享裂变）或嵌入招聘平台。App 在验证阶段绝对不做。
7.2 是否接大模型 → 核心智能必须接，但 V0 先用规则
纯规则做不出"动态维度 + 有依据点评 + 风险推理"。但 V0 用规则跑通闭环最划算：零 API 依赖、快、验证"用户到底要不要看能力地图"。资源够可 V0 直接挂轻量模型。
7.3 接入什么模型 → 国产为主：DeepSeek / 通义千问
选型看四点：中文能力、结构化 JSON 输出（schema 遵循）、成本、合规。主力 DeepSeek / 通义千问（便宜、中文好、支持 JSON mode）；备选 GLM-4 / Kimi（长上下文适合长简历）。海外模型能力强但国内访问/合规/成本吃亏。
7.4 密钥与调用 → 轻后端代理，密钥不进前端
前端只传"简历文本 + 维度定义"给后端，后端持密钥调模型。支持用户自带 Key 降本。这是产品的安全与成本基线，从 V1 设计之初就固化。

8数据安全与隐私（红线）

红线原则：简历 = 敏感个人信息（PII）。处理不当既违法又毁信任。
要求实现
最小收集仅解析生成地图所需字段，不收集无关信息
传输加密全链路 HTTPS；后端内网调用 LLM
不留存 / 可删除报告默认不持久化；用户可一键删除；聚合数据去标识化
本地优先（V0）规则引擎可在浏览器端运行，简历不出本机
明示告知上传前清晰告知"用于生成能力地图、是否留存、如何删除"
合规遵循《个人信息保护法》；若做行业基准需用户单独授权
安全不只是合规底线，更是产品信任卖点：在竞品普遍"上传即留存"的当下，"本地处理 / 不留存 / 可删除"本身就是差异化。对外宣传应突出这一点。

9非功能性需求

类别要求
性能V0 规则生成 ≤ 2s；V1 LLM 生成 ≤ 15s（带 loading 与超时兜底）
兼容Chrome / Edge / Safari 主流版本；支持 PDF / .docx 简历
可用性上传失败 / 解析失败有明确指引，不白屏；空结果有兜底模板
成本V1 单次 LLM 调用成本可控（选小参数模型 + 精简 prompt）
可观测记录生成成功/失败、耗时、模型降级次数，指导迭代

11风险与边界

风险等级应对
评估主观性 / 误判中标注"仅供参考"；V1 用 LLM 提升语境理解；提供手动微调
模型幻觉（V1）中严格 schema 校验；规则兜底；关键结论可溯源到原文
隐私合规高见 §8 红线；本地优先 + 不留存 + 明示授权
需求伪命题中V0 即用规则验证"用户是否真要"，不行则止损
维度库维护成本中维度配置化（YAML/JSON），新增岗位低成本
边界声明：本产品输出不是求职建议、不是背调结论，而是"基于简历文本的自动化初评"，所有结论应结合用户主观判断使用。
